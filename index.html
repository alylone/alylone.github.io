51 汇编总结

1、汇编基础知识

汇编语言种类

汇编语言主要取决于它们所基于的处理器架构。不同的处理器架构有不同的汇编语言。一些常见的汇编语言类型包括：

1.x86/x64汇编语言：用于Intel和AMD处理器，这是在个人计算机上最常见的架构。

2.ARM汇编语言：用于ARM处理器，广泛应用于移动设备如智能手机和平板电脑。

3.MIPS汇编语言：用于MIPS处理器，常见于嵌入式系统和早期的个人电脑。

4.PowerPC汇编语言：曾经用于旧款苹果电脑和某些游戏机。

5.6502汇编语言：用于6502微处理器，曾经在早期的个人电脑和游戏机中广泛使用。

6.SPARC汇编语言：用于SPARC处理器，主要用于某些服务器和高性能计算机。

这些只是一些例子，还有许多其他类型的处理器架构，每种架构都有其特定的汇编语言。汇编语言通常非常底层，直接对应处理器的机器指令集。

8051汇编语言

51单片机（也称为8051单片机）使用的是一种特定于其处理器架构的汇编语言，通常被称为8051汇编语言。这种汇编语言是专门为8051系列微控制器设计的，与上述提到的x86、ARM等汇编语言有显著的不同。

8051微控制器由Intel公司在1980年代推出，其汇编语言直接对应8051微控制器的指令集。由于8051系列微控制器在嵌入式系统设计中的广泛应用，其汇编语言在嵌入式编程领域仍然非常重要。

汇编器

汇编器（Assembler）是一种将汇编语言代码转换成机器语言代码的程序。不同的汇编器支持不同的汇编语言，通常特定于某个处理器架构。以下是一些常见的汇编器：

1.NASM (Netwide Assembler): 非常流行的汇编器，主要用于Intel x86架构。

2.MASM (Microsoft Macro Assembler): 由微软开发，专用于Windows平台上的x86架构。

3.GAS (GNU Assembler): GNU项目的一部分，支持多种处理器架构，包括x86和ARM。

4.FASM (Flat Assembler): 一个小巧且高效的汇编器，主要用于x86架构。

5.TASM (Turbo Assembler): 由Borland开发，适用于x86架构，与MASM类似。

6.ARM Assembler: 针对ARM处理器的汇编器，通常作为ARM开发工具链的一部分。

7.Keil Assembler: 主要用于ARM和8051微控制器的嵌入式系统开发。

8.AVR Assembler: 用于Atmel AVR系列微控制器的汇编器。

这些汇编器覆盖了多种处理器架构和平台。选择哪个汇编器取决于目标处理器、操作系统以及特定的开发需求。

汇编文件的后缀

.asm：在Windows环境中使用MASM汇编器时常用的文件扩展名。
.s：在Unix或Linux环境中使用GCC或Clang汇编器时常用的文件扩展名。
.nasm：在NASM汇编器中使用的文件扩展名。
.a51：在Keil C51开发环境中使用的文件扩展名。
.inc：在汇编语言中通常用于包含头文件或宏定义。

汇编为什么可以移植性差
汇编语言的可移植性差主要是因为它是一种非常底层的编程语言，直接与特定硬件架构的指令集紧密相关。以下是几个关键原因：

1.硬件依赖性：汇编语言指令直接映射到特定处理器的机器指令上。不同的处理器架构（如x86、ARM、MIPS）拥有不同的指令集，因此为一个架构编写的汇编代码不能直接在另一个架构上运行。

2.操作系统依赖性：汇编语言编写的程序通常需要直接与操作系统交互，使用特定于操作系统的系统调用和约定。这意味着即使在相同的处理器架构上，不同操作系统之间的汇编代码也可能不兼容。

3.低级别的控制和优化：汇编语言允许程序员进行针对特定硬件的低级别优化。这种优化通常是特定于某种硬件的，导致代码与其他硬件不兼容。

4.缺乏抽象：高级语言如C、Java或Python提供了很多抽象，隐藏了底层硬件的细节。而汇编语言几乎不提供这样的抽象，使得代码直接反映了底层硬件的特性。

总之，汇编语言的代码通常需要根据目标硬件和操作系统进行调整或完全重写，以适应不同的架构和平台，这大大限制了它的可移植性。

HEX文件

烧录时的hex文件是什么

HEX文件是一种用于嵌入式系统编程的文件格式。这种文件通常包含机器语言代码，用于微控制器或其他嵌入式设备。HEX文件的内容通常表示为十六进制数字，这是一种便于人类阅读和理解的计算机程序和数据的表示方式。

在嵌入式系统开发中，HEX文件通常用于以下几个方面：

1.固件编程：开发者会将编写的源代码编译成机器代码，然后再转换成HEX文件格式。这个文件随后可以通过编程器上传到微控制器或其他嵌入式设备。

2.固件更新：在需要更新嵌入式设备上的软件时，HEX文件被用来替换或更新设备上现有的固件。

3.调试和测试：开发者可能使用HEX文件来测试和调试嵌入式系统的各个部分，确保它们按预期工作。

HEX文件的格式和内容可能根据使用的微控制器和编程环境而有所不同，但它们普遍作为嵌入式系统开发中的重要组成部分。

函数结束时，有关变量的问题

在C51（即8051微控制器）的编程中，函数结束时，unsigned int 类型变量不被释放的原因通常与变量的存储类别和内存管理有关。以下是一些可能的原因：

  1. 全局变量：如果unsigned int类型的变量是全局变量，它在程序的整个生命周期内都存在，并不会在函数结束时被释放。全局变量通常存储在静态内存区域中。

  2. 静态局部变量：如果在函数中声明了static unsigned int类型的局部变量，它的生命周期与程序相同，即使函数结束，该变量也不会被释放。静态局部变量在函数首次调用时被初始化，以后每次调用函数时都会保持上一次调用后的值。

  3. 非静态局部变量：如果变量是非静态的局部变量，通常情况下，它应该在函数返回时被释放。但由于C51编译器和8051架构的特点，这些变量可能存储在寄存器或内部RAM中，而这些内存区域在函数调用结束后不会被明确“释放”，只是下一次函数调用时可以被覆盖或重新分配。

详细解释

  • 全局变量：全局变量在程序加载时被分配内存，并在程序结束时释放。
  • 静态局部变量：静态局部变量在函数首次调用时分配内存，并在程序结束时释放。它们保持值在函数调用之间。
  • 自动局部变量：自动局部变量（即没有static修饰的局部变量）在函数调用时分配内存，在函数返回时应该释放。然而，在嵌入式系统（如8051）中，内存管理和变量释放机制并不像在复杂操作系统中那样明确和自动化。局部变量可能仍然保留在栈或寄存器中，直到被新数据覆盖。

总结

在C51编程中，由于内存管理的限制和具体实现细节，函数结束时unsigned int类型的局部变量不会明确地“被释放”，它们的内存可能仍然保持分配状态，直到被新数据覆盖或重新分配。这是嵌入式系统和低级编程环境中的常见情况。

什么是 PS1？

PS1 (Prompt String 1) 是一个环境变量，它定义了你在终端中看到的命令行提示符的样式和内容。当你准备输入命令时，PS1 的内容就会显示出来。

PS1 的基本结构与常用转义符

PS1 的值是一个字符串，其中包含特殊的转义序列，这些序列会被 Zsh 解释为特定的信息（例如当前目录、用户名等）或颜色。

以下是一些常用的转义符：

  • %n: 用户名
  • %m: 主机名（到第一个 . 之前的部分）
  • %M: 完整主机名
  • %~: 当前工作目录，使用 ~ 代替 $HOME
  • %c: 当前工作目录的 basename（只显示最后一个目录名）
  • %d: 当前日期（星期几 月份 日期）
  • %t: 当前时间（HH:MM:SS）
  • %*: 当前时间（HH:MM AM/PM）
  • %#: 如果是 root 用户则显示 #，否则显示 %
  • %?: 上一个命令的退出状态码
  • %(?,true_text,false_text): 条件表达式，如果 ? 为真则显示 true_text，否则显示 false_text。例如：%(?,%F{red}%?,) 表示如果上一个命令失败则显示红色退出码。

添加字体效果

Zsh 使用特殊的转义序列来定义颜色。基本格式是 %F{颜色} 和 %f。

  • %F{颜色}: 设置前景色。常见的颜色有 black, red, green, yellow, blue, magenta, cyan, white。你也可以使用 0-255 的数字来表示 256 色。
  • %f: 重置前景色。
  • %K{颜色}: 设置背景色。用法同前景色。
  • %k: 重置背景色。
  • %B: 开启粗体
  • %b: 关闭粗体
  • %U: 开启下划线
  • %u: 关闭下划线

Paru

Paru 是一款为 Arch Linux 及其衍生发行版设计的命令行工具，其核心定位是 Arch 用户软件库 (AUR) 的助手程序 (AUR Helper)。它的主要目标是自动化管理 AUR 软件包的安装、更新与移除过程，并将其与官方仓库的包管理无缝集成。

Arch Linux 的软件体系由两部分构成：

  1. 官方软件仓库：包含由 Arch Linux 官方团队维护的、经过预编译的二进制软件包，通过包管理器 pacman 进行管理。
  2. Arch 用户软件库 (AUR)：一个由社区驱动的软件库，它不包含预编译的软件包，而是提供名为 PKGBUILD 的构建脚本。用户需要手动下载脚本，解决依赖，编译源码，最后打包并安装。

在 Vim 中进行复制（yank）和粘贴（put）时，存在“随意储存器”（更准确的说法是无名寄存器 Unnamed Register）和“系统储存器”（即剪贴板寄存器 Clipboard Registers）的概念。

1. Vim 的寄存器 (Registers)

Vim 不像大多数其他编辑器那样只有一个剪贴板。它有多个独立的“储存器”，称为寄存器。当你复制（yank）或删除（delete）文本时，这些文本会被存放在某个寄存器中。当你粘贴（put）时，Vim 会从某个寄存器中取出文本。

每个寄存器都有一个名称，通常是一个字符（如 a, b, +, * 等）或一个数字。你可以通过在复制、删除或粘贴命令前加上双引号和寄存器名称来指定使用哪个寄存器。例如：

  • "ayy: 将当前行复制到寄存器 a 中。
  • "bdw: 删除一个单词并将其存入寄存器 b 中。
  • "ap: 将寄存器 a 中的内容粘贴到光标后。

如果你在复制、删除或粘贴时没有指定寄存器，Vim 就会使用默认的寄存器。这就是“无名寄存器”和“系统剪贴板寄存器”发挥作用的地方。

2. 无名寄存器 (Unnamed Register) - ""

  • 概念： 这是 Vim 的默认寄存器。它的名称是两个双引号 ""。
  • 工作方式：
    • 当你使用 y (yank)、d (delete)、c (change) 等命令且没有指定寄存器时，被操作的文本会自动存入无名寄存器 "" 中。
    • 当你使用 p (put) 或 P 命令且没有指定寄存器时，Vim 会自动从无名寄存器 "" 中取出文本进行粘贴。

3. 剪贴板寄存器 (Clipboard Registers) - "* 和 "+

  • 概念： 这两个寄存器是 Vim 用来与操作系统的剪贴板进行交互的。它们是 Vim 中的“系统储存器”。
  • "* 寄存器： 通常对应于 X Window System (Linux/Unix) 中的选择剪贴板 (selection clipboard)。当你用鼠标选中一段文本时，这段文本通常会自动进入这个剪贴板。在某些系统或配置下，"* 也可能与主剪贴板关联。
  • "+ 寄存器： 这是最重要的一个，它通常对应于操作系统的主剪贴板 (clipboard)。这个剪贴板就是你平时使用 Ctrl+C / Ctrl+V (Windows/Linux) 或 Cmd+C / Cmd+V (macOS) 进行复制粘贴时所使用的剪贴板。
  • 工作方式：
    • 你可以显式地将文本复制到系统剪贴板：
      • "*yy: 将当前行复制到选择剪贴板。
      • "+yy: 将当前行复制到主剪贴板（这就是将 Vim 内容复制到系统其他应用）。
    • 你可以显式地从系统剪贴板粘贴文本：
      • "*p: 从选择剪贴板粘贴。
      • "+p: 从主剪贴板粘贴（这就是将系统其他应用的内容粘贴到 Vim）。

4. 无名寄存器与剪贴板寄存器的交互 (clipboard 选项)

Vim 有一个 clipboard 选项，可以控制无名寄存器 "" 是否以及如何与系统剪贴板寄存器 "* 和 "+ 自动同步。

  • set clipboard= (默认或空): 无名寄存器 "" 不与系统剪贴板自动同步。你必须显式地使用 "* 或 "+ 寄存器才能与系统剪贴板交互。
  • set clipboard=unnamed: 当你复制或删除到无名寄存器 "" 时，文本会同时被复制到 "* 寄存器。当你从无名寄存器 "" 粘贴时，Vim 会先尝试从 "* 寄存器获取内容。
  • set clipboard=unnamedplus: 这是最常用的设置，尤其是在希望 Vim 与系统剪贴板无缝集成时。当你复制或删除到无名寄存器 "" 时，文本会同时被复制到 "+ 寄存器。当你从无名寄存器 "" 粘贴时，Vim 会先尝试从 "+ 寄存器获取内容。
  • set clipboard=unnamed,unnamedplus: 同时启用 unnamed 和 unnamedplus 的行为。

总结：

  • 无名寄存器 ("") 是 Vim 内部的默认临时储存区，内容随最近操作变化，是你使用 y, d, p 等命令不指定寄存器时的目标/来源。它可能被你称为“随意储存器”。
  • 剪贴板寄存器 ("*, "+") 是 Vim 与操作系统剪贴板交互的接口，"+ 通常对应你日常使用的系统剪贴板。它们是你进行跨应用复制粘贴的“系统储存器”。
  • clipboard 选项控制着无名寄存器是否自动与系统剪贴板同步。设置 set clipboard=unnamedplus 可以让你在 Vim 中使用 y 复制的内容直接在系统其他应用中粘贴 (Ctrl+V)，以及在系统其他应用中复制的内容 (Ctrl+C) 直接在 Vim 中使用 p 粘贴，实现无缝集成。

1、Docker 客户端（Docker Client）

Docker 客户端是用户与 Docker 守护进程交互的命令行界面（CLI）。它是用户与 Docker 系统的主要交互方式，用户通过 Docker CLI 发出命令，这些命令被发送到 Docker 守护进程，由守护进程执行相应的操作。

  • 功能：允许用户使用命令与 Docker 守护进程通信，如创建容器、构建镜像、查看容器状态等。
  • 交互方式：Docker 客户端与 Docker 守护进程之间通过 REST API 或 Unix 套接字通信。常用的命令行工具是 docker，通过它，用户可以发出各种 Docker 操作命令。

常用命令：

  • docker run：运行容器。
  • docker ps：列出正在运行的容器。
  • docker build：构建 Docker 镜像。
  • docker exec：在容器中执行命令。

2、Docker 守护进程（Docker Daemon）

Docker 守护进程（通常是 dockerd）是 Docker 架构的核心，负责管理容器生命周期、构建镜像、分发镜像等任务。

守护进程通常以后台进程的方式运行，等待来自 Docker 客户端的 API 请求。

功能：

  • 启动和停止容器。
  • 构建、拉取和推送镜像。
  • 管理容器的网络和存储。
  • 启动、停止、查看容器日志等。
  • 与 Docker 注册表进行通信，管理镜像的存储与分发。

Docker 守护进程监听来自 Docker 客户端的请求，并且通过 Docker API 执行这些请求。守护进程将负责容器、镜像等 Docker 对象的管理，并根据请求的参数启动容器、删除容器、修改容器配置等。

5、Docker in Docker？

在 Docker 容器内部使用 sudo service docker start 来启动 Docker 守护进程（即实现 Docker in Docker，简称 DinD）通常是不可行的，原因如下：

  1. 服务管理器：service 命令依赖于系统的服务管理器（如 systemd 或 SysVinit）。然而，在许多 Docker 镜像中，尤其是基于 Alpine Linux 或其他精简版 Linux 发行版的镜像，可能不包含完整的服务管理器。即使包含，服务管理器也未必在容器内以正常方式运行。
  2. Docker in Docker 实践：要在 Docker 容器内部运行另一个 Docker 实例，通常的做法是使用 Docker 官方提供的 Docker in Docker (docker:dind) 镜像，或者将宿主机的 Docker 套接字挂载到容器内部，而不是尝试在容器内启动 Docker 守护进程。

解决方案：

1. 使用 Docker 官方的 DinD 镜像

这种方法不需要在容器内部启动 Docker 服务。你可以直接使用 Docker 提供的 docker:dind 镜像运行一个容器，这个镜像已经为运行 Docker 守护进程进行了配置。

注意：使用 --privileged 标志会给容器提供较高的权限，这可能带来安全风险，请谨慎使用。

2. 将宿主机的 Docker 套接字挂载到容器

这种方法允许容器内的进程通过宿主机的 Docker 守护进程来启动和管理 Docker 容器。

这条命令创建了一个新的容器，并将宿主机的 /var/run/docker.sock 文件挂载到容器内部。这样，容器内部的 Docker 客户端就可以控制宿主机的 Docker 守护进程了。

总结:均为在宿主机创建docker时，使用特殊的创建方法，而不是在进入容器后进行的操作，针对AutoDl的情况，没有很好的处理办法

6、与传统虚拟机对比

二者的不同：

  • VM(VMware)在宿主机器、宿主机器操作系统的基础上创建虚拟层、虚拟化的操作系统、虚拟化的仓库，然后再安装应用；
  • Container(Docker容器)，在宿主机器、宿主机器操作系统上创建Docker引擎，在引擎的基础上再安装应用。

与传统VM特性对比：
作为一种轻量级的虚拟化方式，Docker在运行应用上跟传统的虚拟机方式相比具有显著优势：- Docker 容器很快，启动和停止可以在秒级实现，这相比传统的虚拟机方式要快得多。

  • Docker 容器对系统资源需求很少，一台主机上可以同时运行数千个Docker容器。
  • Docker 通过类似Git的操作来方便用户获取、分发和更新应用镜像，指令简明，学习成本较低。
  • Docker 通过Dockerfile配置文件来支持灵活的自动化创建和部署机制，提高工作效率。
  • Docker 容器除了运行其中的应用之外，基本不消耗额外的系统资源，保证应用性能的同时，尽量减小系统开销。
  • Docker 利用Linux系统上的多种防护机制实现了严格可靠的隔离。从1.3版本开始，Docker引入了安全选项和镜像签名机制，极大地提高了使用Docker的安全性。

如何Pull Requests

Pull Request 是一种请求，用于将你所做的工作（你的代码更改）合并（pull）到项目的另一个分支（通常是主分支）中。

参考文章:https://www.freecodecamp.org/chinese/news/how-to-make-your-first-pull-request-on-github/

想象一下：你正在为某个项目贡献代码。为了不直接修改主代码库（可能导致问题），你会：

  1. 复制（Fork） 项目到你自己的账户下（在 GitHub 上是 Fork）。
  2. 克隆（Clone） 你复制的项目到本地。
  3. 创建新分支： 基于项目的稳定版本（例如 main 或 master 分支）创建一个新的、独立的分支，所有的开发工作都在这个分支上进行。
  4. 编码和提交： 在这个新分支上进行代码修改、添加新功能或修复 bug，并多次提交（commit）你的更改。
  5. 推送（Push） 你的新分支及其上的所有提交到你自己的远程代码库。
  6. 发起 Pull Request： 在 GitHub 等平台上，你会发起一个 Pull Request，请求项目维护者将你新分支上的代码“拉取”（pull）到他们的主分支中。
